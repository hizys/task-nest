# CI/CD 深入学习说明

本文档用于理解：

- CI/CD 是什么。
- 自动化流水线是什么。
- 滚动发布、蓝绿发布、灰度发布是什么。
- 回滚策略怎么设计。
- dev、test、staging、prod 环境怎么隔离。
- TaskNest 可以怎么设计发布流程。

## 1. 先看结论

CI 是：

```text
Continuous Integration
持续集成
```

核心目标：

```text
代码一提交，就自动检查代码有没有问题。
```

CD 是：

```text
Continuous Delivery / Continuous Deployment
持续交付 / 持续部署
```

核心目标：

```text
让代码可以稳定、可重复、可回滚地发布到环境。
```

自动化流水线就是：

```text
把拉代码、测试、构建、打包、部署、验证串成自动流程。
```

## 2. 为什么需要 CI/CD

没有 CI/CD 时，部署依赖人工：

```text
手动登录服务器
手动 git pull
手动编译
手动复制文件
手动重启服务
手动看日志
```

问题：

```text
容易漏步骤
容易敲错命令
每次发布不一致
失败不好定位
回滚麻烦
新人不敢部署
过程不可追踪
```

有 CI/CD 后：

```text
流程标准化
操作自动化
结果可追踪
失败可定位
版本可回滚
多人协作更安全
```

## 3. CI 通常做什么

CI 阶段通常包括：

```text
拉取代码
安装依赖
代码格式检查
静态扫描
运行单元测试
运行集成测试
构建应用
构建 Docker 镜像
生成测试报告
```

TaskNest 可以先从这些开始：

```bash
go test ./...
go vet ./...
go build -o task-nest .
```

## 4. CD 通常做什么

CD 阶段通常包括：

```text
构建 Docker 镜像
打版本号
推送镜像仓库
部署到 test
部署到 staging
审批后部署 prod
执行健康检查
观察监控
失败时回滚
```

完整流程：

```text
开发提交代码
   |
   v
CI 自动测试
   |
   v
构建镜像
   |
   v
部署测试环境
   |
   v
预发布验证
   |
   v
生产发布
   |
   v
观察监控并决定是否回滚
```

## 5. 自动化流水线是什么

流水线是多个阶段组成的自动流程。

常见阶段：

```text
Checkout  拉代码
Test      运行测试
Build     编译
Package   构建镜像
Push      推送镜像
Deploy    部署
Verify    验证
Notify    通知
```

Jenkinsfile 示例：

```groovy
pipeline {
    agent any

    stages {
        stage('Test') {
            steps {
                sh 'go test ./...'
            }
        }

        stage('Build') {
            steps {
                sh 'go build -o task-nest .'
            }
        }

        stage('Docker Build') {
            steps {
                sh 'docker build -t task-nest:${BUILD_NUMBER} .'
            }
        }
    }
}
```

生产不要只用 `latest`。

推荐用：

```text
task-nest:v1.2.0
task-nest:commit-a1b2c3d
task-nest:build-123
```

这样更容易回滚。

## 6. 滚动发布

滚动发布是逐个替换实例。

假设线上有 4 个实例：

```text
v1, v1, v1, v1
```

发布过程：

```text
v2, v1, v1, v1
v2, v2, v1, v1
v2, v2, v2, v1
v2, v2, v2, v2
```

优点：

```text
实现简单
资源消耗少
K8s 默认支持
适合日常发布
```

缺点：

```text
发布过程中新旧版本共存
新旧版本不兼容时可能出问题
```

## 7. 蓝绿发布

蓝绿发布是准备两套环境。

```text
蓝色环境：当前线上 v1
绿色环境：新版本 v2
```

发布前：

```text
用户流量 -> 蓝色环境
绿色环境先部署好但不接流量
```

验证通过后：

```text
用户流量 -> 绿色环境
```

如果有问题：

```text
用户流量 -> 蓝色环境
```

优点：

```text
切换快
回滚快
新版本可提前验证
```

缺点：

```text
需要两套环境
资源成本高
数据库变更要特别小心
```

## 8. 灰度发布

灰度发布也叫金丝雀发布。

核心思想：

```text
先让一小部分用户使用新版本，确认没问题后逐步扩大。
```

例如：

```text
1% 用户 -> v2
5% 用户 -> v2
20% 用户 -> v2
50% 用户 -> v2
100% 用户 -> v2
```

灰度维度：

```text
用户 ID
地区
租户
请求 Header
App 版本
流量比例
内部员工
```

优点：

```text
风险低
影响范围小
能用真实流量验证
```

缺点：

```text
实现复杂
需要网关或流量控制
需要监控支持
新旧版本兼容要求高
```

## 9. 三种发布方式怎么选

```text
日常小改动：滚动发布
重要版本切换：蓝绿发布
高风险功能验证：灰度发布
```

对比：

| 发布方式 | 优点 | 缺点 |
| --- | --- | --- |
| 滚动发布 | 简单、省资源 | 新旧版本共存 |
| 蓝绿发布 | 切换和回滚快 | 需要两套环境 |
| 灰度发布 | 风险最低 | 实现复杂 |

## 10. 回滚是什么

回滚是新版本出问题后，恢复到上一个稳定版本。

例如：

```text
当前 v2 创建任务大量报错
回滚到 v1
```

回滚不是失败，而是保障稳定性的必要能力。

上线前要提前写清楚：

```text
如何发布
如何验证
如何观察
如何回滚
什么条件触发回滚
谁负责回滚
```

## 11. 常见回滚方式

代码回滚：

```text
把代码恢复到旧版本，重新构建部署。
```

镜像回滚：

```text
task-nest:v1.0.1 -> task-nest:v1.0.0
```

K8s 回滚：

```bash
kubectl rollout undo deployment/task-nest
```

配置回滚：

```text
如果是配置改错，只回滚配置。
```

数据库回滚：

```text
最复杂也最危险。
删除字段、修改字段类型、清理数据都要非常谨慎。
```

数据库变更建议：

```text
只做向前兼容
先加字段，后改代码
稳定后再清理旧字段
重要数据操作前备份
```

## 12. 多环境隔离

常见环境：

```text
dev      开发环境
test     测试环境
staging  预发布环境
prod     生产环境
```

为什么要隔离：

```text
开发不能影响真实用户
测试需要稳定环境
上线前要模拟生产
生产环境必须稳定
```

要隔离：

```text
配置
数据库
权限
域名
机器资源
日志
监控
密钥
```

生产环境原则：

```text
不要临时试验
不要直接改数据
不要跳过流程
不要没有回滚方案就上线
```

## 13. TaskNest 发布示例

假设新增任务优先级 `priority` 字段。

开发：

```text
创建 feature/todo-priority 分支
修改 Todo model
修改创建接口
修改更新接口
增加数据库字段
本地测试
提交代码
```

CI：

```bash
go test ./...
go vet ./...
go build
```

构建镜像：

```text
task-nest:v1.3.0
```

部署 test：

```text
验证创建任务可传 priority
验证不传 priority 有默认值
验证旧任务不报错
```

部署 staging：

```text
验证数据库迁移
验证配置
验证日志和监控
验证核心接口
```

发布 prod：

```text
滚动发布
观察错误率、P95、P99、日志
异常则回滚镜像
```

## 14. 学习路线

建议顺序：

1. 理解 CI 和 CD。
2. 理解流水线阶段。
3. 学会 `go test ./...` 接入流水线。
4. 学会构建 Docker 镜像。
5. 理解 dev/test/staging/prod。
6. 理解滚动、蓝绿、灰度。
7. 理解回滚策略。
8. 学会发布后观察监控。
