# AI 编程助手核心概念学习说明

本文档用于理解 AI 编程助手里的几个常见概念：

- Skill 是什么。
- `AGENTS.md`、`CLAUDE.md`、`agents/` 这类文件有什么作用。
- 插件是什么。
- Plan 模式、Subagent 模式、Team 模式分别是什么。
- 为什么长会话需要压缩。
- 如果不压缩并达到模型上下文上限，会发生什么。

## 1. 先看结论

可以先用一句话理解这些概念：

```text
Skill       = 给 AI 增加一套特定任务的操作说明
Agent 文件  = 给 AI 看的项目说明书
Plugin      = 把工具、技能、连接器打包成一个扩展能力
Plan 模式   = 先讨论方案，再执行修改
Subagent    = 把子任务交给专门的 AI 分工处理
Team 模式   = 多个 AI 角色协作完成一个更复杂目标
会话压缩    = 把长上下文总结成短上下文，避免超过模型记忆窗口
```

它们不是业务代码的一部分，而是帮助 AI 更好理解项目、调用工具、拆分任务和保持长时间工作的辅助机制。

## 2. Skill 是什么

Skill 可以理解成 AI 的一份专项操作手册。

普通 AI 助手默认知道很多通用知识，但遇到某些具体任务时，需要按固定流程做事。例如：

- 生成图片时，要遵守图片生成工具的流程。
- 创建 Word 文档时，要生成 `.docx`，再渲染检查排版。
- 处理 GitHub PR 评论时，要先读取评论，再定位代码，再修改。
- 创建新的 Codex skill 时，要按规定目录生成 `SKILL.md`。

这些专项流程就可以写成 Skill。

一个 Skill 通常包含：

```text
SKILL.md        说明这个 skill 什么时候使用、怎么使用
scripts/        可选，放辅助脚本
templates/      可选，放模板文件
assets/         可选，放资源文件
references/     可选，放参考资料
```

简单理解：

```text
没有 Skill：
AI 依靠通用经验做事。

有 Skill：
AI 按这类任务的专用流程做事。
```

Skill 的价值是让 AI 在特定任务上更稳定、更一致，也更少漏步骤。

## 3. AGENTS.md、CLAUDE.md 和 agents/ 文件的作用

在很多 AI 编程工具里，项目根目录可能会出现类似文件：

```text
AGENTS.md
CLAUDE.md
.claude/
agents/
```

这些文件通常是给 AI 编程助手看的项目说明，不是给程序运行时读取的业务配置。

它们常见用途包括：

- 告诉 AI 项目如何启动。
- 告诉 AI 项目如何测试。
- 告诉 AI 代码风格和目录规范。
- 告诉 AI 哪些文件不要改。
- 告诉 AI 提交代码前要运行哪些命令。
- 告诉 AI 某些业务概念怎么理解。
- 给不同 AI 助手配置不同的行为规则。

例如一个 `AGENTS.md` 可能会写：

````markdown
# 项目说明

## 启动

```bash
go run .
```

## 测试

```bash
go test ./...
```

## 约定

- Controller 只处理 HTTP 请求参数和响应。
- Service 负责业务逻辑。
- Model 只放数据结构。
- 不要提交本地数据库文件 `tasknest.db`。
````

### AGENTS.md

`AGENTS.md` 通常是比较通用的 AI 代理说明文件。

它可以被支持该约定的 AI 编程助手读取，用来理解当前仓库的工作方式。

适合放：

- 项目结构说明。
- 构建、测试、运行命令。
- 代码风格。
- 分层规则。
- 安全边界。
- 常见坑。

### CLAUDE.md

`CLAUDE.md` 通常是 Claude Code 这类工具使用的项目说明文件。

它和 `AGENTS.md` 的思路类似，只是面向的工具不同。

如果一个项目同时要给多个 AI 工具使用，可能会同时存在：

```text
AGENTS.md    给通用 agent 读取
CLAUDE.md    给 Claude Code 读取
```

内容可以相似，但也可以根据工具特性做差异化说明。

### agents/ 目录

`agents/` 目录通常用于放多个 agent 的角色配置。

例如：

```text
agents/
├── backend.md
├── frontend.md
├── reviewer.md
└── tester.md
```

每个文件可以描述一个专门角色：

```text
backend.md   专门处理后端代码
frontend.md  专门处理前端页面
reviewer.md  专门做代码审查
tester.md    专门补测试和跑测试
```

这类文件的核心作用是：让不同 AI 角色有不同关注点。

## 4. 插件是什么

插件是给 AI 助手扩展能力的包。

如果说 Skill 是“操作说明”，那么插件通常更像“能力包”。一个插件里面可能包含：

```text
Skills      专项任务说明
Tools       可调用工具
Apps        外部应用连接器
MCP Server  模型上下文协议服务
Assets      模板、脚本、资源
```

例如：

- Browser 插件：让 AI 可以打开和检查浏览器页面。
- GitHub 插件：让 AI 可以读取 PR、Issue、CI 状态。
- Documents 插件：让 AI 可以创建和检查 Word 文档。
- Spreadsheets 插件：让 AI 可以处理表格文件。

简单理解：

```text
Skill 重点是告诉 AI 怎么做。
Tool 重点是让 AI 能实际执行某个动作。
Plugin 重点是把一组相关能力打包安装。
```

举个例子：

```text
任务：检查一个本地前端页面有没有样式问题

可能需要：
Browser 插件提供浏览器控制工具
对应 Skill 告诉 AI 如何打开页面、截图、检查布局
AI 根据检查结果修改代码
```

## 5. Plan 模式是什么

Plan 模式可以理解成“先规划，后执行”。

在这种模式下，AI 更偏向于：

- 先阅读需求。
- 先分析现有代码。
- 先列出修改计划。
- 必要时向用户确认关键选择。
- 确认后再真正改文件、跑命令或执行高影响操作。

适合使用 Plan 模式的场景：

- 需求还不清楚。
- 修改范围较大。
- 有多种实现方案。
- 改动可能影响架构。
- 用户想先学习思路，而不是马上改代码。

例如：

```text
用户：我要把 TaskNest 从 SQLite 改成 MySQL，先帮我分析方案。

Plan 模式下：
AI 应该先说明要改哪些地方、风险是什么、需要哪些配置，
而不是直接开始大规模改代码。
```

Plan 模式的关键点不是“不干活”，而是先把要做什么说明白。

## 6. Subagent 模式是什么

Subagent 可以理解成“把一个子任务交给另一个专门的 AI 代理”。

主 Agent 负责整体目标和最终决策，Subagent 负责某个局部任务。

例如用户提出：

```text
帮我检查这个项目的接口设计、测试覆盖和文档完整性。
```

主 Agent 可以拆成几个子任务：

```text
Subagent A：检查接口设计
Subagent B：检查测试覆盖
Subagent C：检查文档完整性
```

然后主 Agent 汇总结果，给出最终建议。

Subagent 的价值是：

- 并行处理多个方向。
- 让每个子任务更聚焦。
- 减少一个 Agent 同时关注太多细节。
- 对复杂任务做更清晰的分工。

但 Subagent 不是越多越好。任务很小时，直接由一个 Agent 完成反而更简单。

## 7. Team 模式是什么

Team 模式可以理解成“多个 AI 角色协作”。

它和 Subagent 有点像，但更强调角色分工和协作流程。

例如一个软件开发 Team 可能包含：

```text
Planner      负责拆解需求和制定方案
Developer    负责写代码
Reviewer     负责代码审查
Tester       负责测试和验证
Documenter   负责补文档
```

Team 模式适合更复杂、更长期的任务，例如：

- 从 0 到 1 做一个完整功能。
- 做一次大型重构。
- 修复复杂线上问题。
- 为一个项目补测试、文档、CI。
- 多轮迭代一个产品原型。

可以这样区分：

```text
Plan 模式：
强调先规划。

Subagent 模式：
强调把子任务分出去。

Team 模式：
强调多个角色持续协作。
```

## 8. 为什么会话需要压缩

AI 模型每次回答时，并不是拥有无限记忆。

它能看到的内容受一个限制影响，叫上下文窗口。

上下文窗口里会放：

- 系统规则。
- 开发者规则。
- 用户历史消息。
- AI 历史回答。
- 工具调用结果。
- 读取过的代码片段。
- 当前问题。

当会话越来越长，上下文就会越来越大。

如果一直不处理，最终会接近模型能读取的上限。

会话压缩的作用是：

```text
把很长的历史对话和工具结果，总结成更短的关键信息。
```

压缩后，AI 不需要重新读取全部历史，也能知道：

- 当前目标是什么。
- 已经做了哪些事。
- 哪些文件改过。
- 哪些测试跑过。
- 还剩什么没做。
- 用户有哪些重要偏好。

简单类比：

```text
不压缩：
把所有聊天记录、所有命令输出、所有代码片段都塞给模型。

压缩：
把前面的过程整理成一份工作交接记录。
```

## 9. 如果不压缩，达到模型上下文上限会怎么样

如果会话不压缩，并且内容超过模型上下文上限，通常会出现几类问题。

### 9.1 无法继续放入更多内容

模型一次能处理的内容有限。

当历史太长时，新的代码、错误日志、需求说明可能放不进去。

结果是：

```text
AI 无法同时看到旧背景和新问题。
```

### 9.2 早期内容被丢掉

一些系统可能会自动裁剪早期对话。

这会导致 AI 忘记：

- 用户之前定过的约束。
- 已经排除过的方案。
- 已经修改过的文件。
- 某些关键错误出现的原因。

这不是模型故意忘，而是早期内容已经不在它当前能看到的上下文里。

### 9.3 回答质量下降

上下文太长时，即使没有硬性报错，模型也更容易：

- 漏看细节。
- 混淆旧方案和新方案。
- 重复已经做过的分析。
- 引用过期信息。
- 给出不一致的建议。

### 9.4 成本和速度变差

上下文越长，每次模型处理的信息越多。

这会让：

- 响应更慢。
- 消耗更多 token。
- 工具调用和总结更容易变长。

所以压缩不是为了偷懒，而是为了让长任务能继续稳定推进。

## 10. 压缩会不会丢信息

会。

压缩本质上是总结，总结就一定会取舍。

所以好的压缩应该保留：

- 当前目标。
- 用户明确要求。
- 关键设计决策。
- 文件修改记录。
- 测试结果。
- 尚未解决的问题。
- 不能重复做的操作。

不重要的信息可以丢掉，例如：

- 重复的命令输出。
- 已经过时的探索过程。
- 不再采用的临时方案。
- 大段无关日志。

因此，长任务里最重要的是把关键决策说清楚，而不是保存每一句聊天原文。

## 11. 这些概念之间的关系

可以用下面这张关系表理解：

| 概念 | 主要解决什么问题 | 类比 |
| --- | --- | --- |
| Skill | 让 AI 按某类任务的专用流程做事 | 操作手册 |
| Agent 文件 | 让 AI 理解当前项目规则 | 项目说明书 |
| Plugin | 给 AI 安装一组工具和能力 | 扩展包 |
| Plan 模式 | 先想清楚再执行 | 方案评审 |
| Subagent 模式 | 把局部任务分给专门代理 | 分工协作 |
| Team 模式 | 多个角色一起推进复杂目标 | 项目小组 |
| 会话压缩 | 让长对话不超过上下文窗口 | 工作交接摘要 |

## 12. 在 TaskNest 项目里可以怎么用

对于当前 TaskNest 项目，可以这样落地。

### 12.1 添加 AGENTS.md

可以在项目根目录添加：

```text
AGENTS.md
```

里面写：

````markdown
# TaskNest Agent Guide

## Run

```bash
go run .
```

## Test

```bash
go test ./...
```

## Project Structure

- `main.go` starts the HTTP server.
- `routes/` registers routes.
- `controllers/` handles HTTP requests and responses.
- `services/` contains business logic.
- `models/` contains data structures.
- `config/` initializes the database.

## Rules

- Keep controller logic thin.
- Put business logic in services.
- Do not commit local database files.
- Run `gofmt` after editing Go files.
````

这样以后 AI 进入这个项目时，会更容易按你的项目规则工作。

### 12.2 添加 CLAUDE.md

如果你使用 Claude Code，也可以添加：

```text
CLAUDE.md
```

内容可以和 `AGENTS.md` 类似。

如果你同时使用多个 AI 工具，可以让：

```text
AGENTS.md    放通用规则
CLAUDE.md    放 Claude Code 专用规则
```

### 12.3 使用 Plan 模式学习新功能

例如你想学习认证功能，可以先问：

```text
先用 Plan 模式帮我设计 TaskNest 的用户登录和 JWT 鉴权，不要改代码。
```

这样适合学习整体思路。

### 12.4 使用 Subagent 或 Team 模式做复杂任务

例如：

```text
帮我从接口设计、数据库设计、代码结构、测试覆盖四个角度审查 TaskNest。
```

这类任务就适合拆给多个角色或多个子代理。

## 13. 最后总结

这些概念的本质都是为了让 AI 更可靠地完成复杂软件任务：

```text
Skill 解决“怎么做某类任务”
Agent 文件解决“这个项目有什么规则”
Plugin 解决“AI 能调用哪些能力”
Plan 模式解决“先想清楚再动手”
Subagent 模式解决“复杂任务拆分”
Team 模式解决“多角色协作”
会话压缩解决“长任务记忆容量有限”
```

学习 AI 编程助手时，不需要一开始就全部用上。

建议顺序是：

1. 先理解 `AGENTS.md` 或 `CLAUDE.md`。
2. 再理解 Skill。
3. 再理解插件。
4. 最后再学习 Plan、Subagent、Team 这类协作模式。
