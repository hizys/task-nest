# 消息队列 MQ 学习说明

本文档用于理解：

- MQ 是什么。
- 为什么需要消息队列。
- 什么是生产者、消费者、队列、Topic。
- MQ 解决什么问题。
- 消息丢失、重复消费、顺序、堆积怎么理解。
- TaskNest 后续可以怎么使用 MQ。

## 1. 先看结论

MQ 是 Message Queue，消息队列。

核心思想是：

```text
把要做的事情写成消息，先放进队列里，再由消费者慢慢处理。
```

没有 MQ：

```text
接口要同步做完所有事情，再返回给用户。
```

有 MQ：

```text
接口只做核心操作，把非核心事情发送成消息，后台异步处理。
```

MQ 主要解决：

```text
异步处理
削峰填谷
系统解耦
失败重试
最终一致性
```

## 2. MQ 的核心角色

```text
Producer  生产者，发送消息
Message   消息，要处理的数据
Queue     队列，保存消息
Topic     主题，发布订阅场景常用
Consumer  消费者，读取并处理消息
Broker    消息服务器
```

常见 MQ：

```text
Kafka
RabbitMQ
RocketMQ
Pulsar
Redis Stream
```

## 3. 为什么需要 MQ

假设创建任务时要做很多事：

```text
1. 写入任务表
2. 写操作日志
3. 发送通知
4. 更新统计数据
5. 推送消息
6. 调用第三方服务
```

如果全部同步执行，接口会变慢。

更好的方式：

```text
创建任务接口：
1. 校验参数
2. 写入任务表
3. 发送 task.created 消息
4. 返回创建成功

后台消费者：
1. 写操作日志
2. 更新统计数据
3. 发送通知
```

这样用户不用等所有附加逻辑完成。

## 4. 异步处理

有些事情不需要立刻完成。

例如：

```text
发送邮件
发送短信
写操作日志
更新统计
生成报表
推送通知
```

这些适合异步。

核心接口先返回成功，后台慢慢处理。

## 5. 削峰填谷

削峰填谷是指：

```text
高峰期请求先进入队列，消费者按能力慢慢处理。
```

例如一秒来了 10 万个请求。

如果都直接写数据库，数据库可能被打爆。

使用 MQ：

```text
请求先写入 MQ
消费者每秒处理 5000 个
慢慢消化队列
```

这能保护数据库和下游服务。

## 6. 系统解耦

没有 MQ 时：

```text
任务服务 -> 通知服务
任务服务 -> 统计服务
任务服务 -> 日志服务
```

任务服务依赖很多服务。

有 MQ 后：

```text
任务服务 -> MQ
通知服务 <- MQ
统计服务 <- MQ
日志服务 <- MQ
```

任务服务只负责发事件，不关心谁消费。

## 7. 点对点和发布订阅

点对点：

```text
一条消息只被一个消费者处理。
```

适合任务分发。

发布订阅：

```text
一条消息可以被多个消费者处理。
```

适合事件通知。

例如：

```text
task.created 事件
通知服务消费
统计服务消费
日志服务消费
```

## 8. 消息丢失

消息可能在这些地方丢：

```text
生产者发送失败
MQ 没有持久化
消费者处理失败但提前确认
```

常见解决方式：

```text
生产者确认
消息持久化
消费者处理成功后再确认
失败重试
死信队列
```

## 9. 重复消费

很多 MQ 更常保证：

```text
至少消费一次
```

也就是说，消息可能重复投递。

所以消费者要做幂等。

幂等是：

```text
同一条消息处理一次和处理多次，最终结果一样。
```

做法：

```text
消息带唯一 message_id
消费前检查是否处理过
数据库唯一索引防重复
使用状态机控制
```

## 10. 消息顺序

有些业务要求顺序。

例如：

```text
先创建任务
再更新任务
最后删除任务
```

如果顺序乱了，会出问题。

解决方式：

```text
同一个业务 ID 的消息进入同一个分区
同一个业务 ID 单线程消费
业务上允许最终一致
```

## 11. 消息堆积

消息堆积是：

```text
生产者发送太快，消费者处理太慢。
```

解决方式：

```text
增加消费者
优化消费逻辑
扩容 MQ
限流生产者
排查下游是否变慢
```

## 12. 死信队列

如果一条消息一直处理失败，可以放入死信队列。

例如：

```text
发送短信失败 3 次
进入死信队列
后续人工排查或补偿
```

死信队列用于避免失败消息一直阻塞正常队列。

## 13. 事务一致性问题

常见问题：

```text
数据库写成功了，但消息发送失败。
消息发送成功了，但数据库写失败。
```

解决思路：

```text
本地消息表
事务消息
Outbox 模式
定时补偿任务
```

本地消息表大概是：

```text
1. 在同一个数据库事务里写业务数据和消息表
2. 后台任务扫描未发送消息
3. 发送到 MQ
4. 发送成功后标记完成
5. 失败则重试
```

## 14. TaskNest 示例

创建任务后发送事件：

```json
{
  "message_id": "msg_001",
  "event_type": "task.created",
  "occurred_at": "2026-06-02T10:00:00+08:00",
  "data": {
    "task_id": 1001,
    "user_id": 1,
    "title": "学习 MQ",
    "status": "pending"
  }
}
```

消费者可以做：

```text
写操作日志
更新任务统计
发送通知
触发提醒
```

任务完成后可以发送：

```text
task.completed
```

用于：

```text
更新完成数量
触发成就系统
发送通知
```

## 15. TaskNest 本地消息表示例

学习阶段可以先不用真实 MQ，用数据库表模拟。

```sql
CREATE TABLE outbox_messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    message_id TEXT NOT NULL UNIQUE,
    event_type TEXT NOT NULL,
    payload TEXT NOT NULL,
    status TEXT NOT NULL,
    retry_count INTEGER NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);
```

创建任务时：

```text
1. 写 todos 表
2. 写 outbox_messages 表
3. 返回成功
```

后台任务：

```text
1. 扫描 pending 消息
2. 执行业务处理
3. 成功改 success
4. 失败增加 retry_count
```

## 16. 学习路线

建议顺序：

1. 理解生产者、消费者、消息、队列。
2. 理解异步、削峰、解耦。
3. 理解消息丢失、重复消费、顺序、堆积。
4. 用本地消息表模拟 MQ。
5. 再学习 RabbitMQ、Kafka 或 Redis Stream。
6. 在 TaskNest 里做任务创建事件。
7. 给消费者加重试和日志。
