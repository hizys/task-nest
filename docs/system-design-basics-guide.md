# 系统设计基础学习说明

本文档用于理解：

- 高可用是什么。
- 高并发是什么。
- 限流、熔断、降级是什么。
- 负载均衡是什么。
- 水平扩容和垂直扩容是什么。
- TaskNest 后续如何逐步演进。

## 1. 先看结论

系统设计不是一开始就追求复杂架构。

它是在思考：

```text
用户变多了怎么办？
某个接口变慢怎么办？
数据库挂了怎么办？
流量突然暴涨怎么办？
外部服务异常怎么办？
服务升级时如何不中断用户？
```

核心概念：

```text
高可用
高并发
限流
熔断
降级
负载均衡
水平扩容
垂直扩容
```

## 2. 高可用是什么

高可用是 High Availability，简称 HA。

意思是：

```text
系统尽可能长时间保持可用。
即使部分机器或服务出问题，也不要让整个系统不可用。
```

如果 TaskNest 只有一台服务器：

```text
用户 -> TaskNest 服务 -> 数据库
```

服务器挂了，系统就不可用。

如果有多台服务：

```text
用户 -> 负载均衡 -> TaskNest A
                  -> TaskNest B
                  -> TaskNest C
```

A 挂了，请求还可以去 B 或 C。

## 3. 为什么需要高可用

真实系统一定会遇到故障：

```text
服务器宕机
程序崩溃
数据库连接失败
网络抖动
磁盘满了
CPU 或内存打满
第三方服务不可用
发布新版本出错
```

高可用目标不是永远不出问题，而是：

```text
局部故障不影响整体
短暂故障自动恢复
严重故障快速切换
```

## 4. 高并发是什么

高并发是同一时间大量请求访问系统。

例如：

```text
每秒 1 个请求
每秒 1000 个请求
每秒 10000 个请求
```

高并发关注：

```text
接口响应时间
数据库连接数
CPU 使用率
内存使用率
慢 SQL
锁竞争
缓存命中率
队列堆积
错误率
```

高并发不是单纯让接口变快，而是让系统在大量请求下仍然稳定。

## 5. 限流是什么

限流是限制请求进入系统的速度。

例如：

```text
同一个 IP 每秒最多请求 10 次
同一个用户每分钟最多创建 60 个任务
整个系统每秒最多处理 1000 个请求
```

超过限制返回：

```text
HTTP 429 Too Many Requests
```

限流的目的：

```text
保护系统，不让异常流量把服务打垮。
```

TaskNest 示例：

```text
POST /api/todos
同一个用户每分钟最多创建 60 个任务。
```

## 6. 熔断是什么

熔断是当某个依赖连续失败时，暂时停止调用它。

例如 TaskNest 未来接入邮件服务：

```text
TaskNest -> 邮件服务
```

如果邮件服务挂了，还一直调用，会导致：

```text
请求等待
goroutine 堆积
接口变慢
TaskNest 自己也被拖垮
```

熔断策略：

```text
邮件服务连续失败 10 次
接下来 30 秒不再调用
直接走失败或降级逻辑
```

核心思想：

```text
发现下游不可用时，先保护自己。
```

## 7. 降级是什么

降级是系统压力大或部分能力不可用时，关闭不重要功能，保证核心功能可用。

TaskNest 核心功能：

```text
创建任务
查询任务
更新任务
删除任务
```

非核心功能：

```text
统计报表
邮件提醒
排行榜
AI 推荐分类
```

压力大时：

```text
任务管理继续可用
统计报表暂时不可用
邮件提醒延迟发送
```

降级目标：

```text
保核心，舍非核心。
```

## 8. 限流、熔断、降级的区别

```text
限流：
请求太多，限制流量进来。

熔断：
下游坏了，暂时不要继续调用。

降级：
系统压力大，关闭不重要功能。
```

例子：

```text
限流：每秒最多 1000 个请求进入 TaskNest。
熔断：邮件服务连续失败后暂时不调用。
降级：压力大时关闭统计报表。
```

## 9. 负载均衡是什么

负载均衡是把请求分发到多台服务器。

没有负载均衡：

```text
用户 -> TaskNest A
```

所有请求都打到 A。

有负载均衡：

```text
用户 -> 负载均衡 -> TaskNest A
                  -> TaskNest B
                  -> TaskNest C
```

常见策略：

```text
轮询
随机
最少连接
权重
IP Hash
```

常见组件：

```text
Nginx
HAProxy
LVS
Kubernetes Service
云负载均衡
```

## 10. 水平扩容

水平扩容是增加机器或实例数量。

例如：

```text
TaskNest A
```

扩成：

```text
TaskNest A
TaskNest B
TaskNest C
```

优点：

```text
承接更多请求
单台挂了不影响整体
适合互联网服务
```

注意：

```text
不能依赖本地内存保存关键状态
本地文件要放共享存储或对象存储
日志要统一收集
配置要统一管理
数据库连接数要控制
```

## 11. 垂直扩容

垂直扩容是提升单台机器配置。

例如：

```text
2 核 4G -> 4 核 8G
4 核 8G -> 8 核 16G
```

优点：

```text
简单直接
不需要大改架构
小系统很实用
```

缺点：

```text
单机有上限
成本越来越高
单点故障仍然存在
```

简单记：

```text
垂直扩容：把一台机器变强。
水平扩容：增加更多机器一起干活。
```

## 12. 常见风险

单点故障：

```text
只有一台应用服务器
只有一个数据库
只有一个 Redis
只有一个网关
```

数据库瓶颈：

```text
慢 SQL
缺少索引
连接数不够
大事务
锁等待
```

没有超时：

```text
外部调用一直等待，拖住资源。
```

没有限流：

```text
突发流量直接打垮服务。
```

没有降级：

```text
非核心功能拖垮核心功能。
```

## 13. TaskNest 示例

任务列表变慢：

```text
给查询字段加索引
接口分页
避免一次返回太多数据
使用缓存
增加服务实例
```

创建任务被刷：

```text
同一 IP 每秒最多 10 次
同一用户每分钟最多创建 60 个任务
超过返回 429
```

邮件服务异常：

```text
任务创建成功
邮件发送失败只记录日志
连续失败后熔断
后台异步重试
```

统计功能压力大：

```text
临时关闭统计报表
保留任务增删改查
```

## 14. 学习路线

建议顺序：

1. 理解高可用和高并发。
2. 理解限流、熔断、降级。
3. 理解负载均衡。
4. 理解水平扩容和垂直扩容。
5. 学会从日志、监控、PProf 定位瓶颈。
6. 给 TaskNest 增加分页、索引、超时。
7. 学习 Nginx 或 K8s Service 做负载均衡。
8. 学习缓存、MQ、监控如何支撑高并发。
