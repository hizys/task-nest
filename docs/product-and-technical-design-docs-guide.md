# 产品文档和技术设计文档学习说明

本文档用于理解软件项目里两类重要文档：

- 产品文档是什么样子的。
- 技术设计文档是什么样子的。
- 技术设计文档应该包含哪些内容。
- 如何结合 TaskNest 项目写一份可用的文档。

## 1. 先看结论

产品文档和技术设计文档关注点不同。

```text
产品文档：
说明要做什么、为什么做、给谁用、用户怎么用。

技术设计文档：
说明怎么实现、接口怎么设计、数据怎么流转、数据库怎么设计、影响范围是什么。
```

可以简单理解：

```text
产品文档面向需求。
技术设计文档面向实现。
```

如果一个需求是“给 TaskNest 增加任务分类功能”，两类文档的关注点是：

```text
产品文档：
用户为什么需要任务分类？
页面或接口要支持哪些操作？
什么情况下算功能完成？

技术设计文档：
要新增哪些接口？
要新增哪些表字段？
Controller、Service、Model 怎么改？
数据从请求到数据库怎么流转？
会影响哪些旧接口？
```

## 2. 产品文档是什么

产品文档通常也叫 PRD。

PRD 的全称是：

```text
Product Requirements Document
产品需求文档
```

它主要回答：

```text
我们要做什么功能？
为什么要做？
谁会使用？
用户怎么使用？
什么情况算做完？
```

产品文档不重点讲代码怎么写，也不重点讲数据库怎么设计。

它更关注用户、场景、流程和规则。

## 3. 产品文档常见结构

一份产品文档通常包含：

```text
1. 文档信息
2. 需求背景
3. 目标和收益
4. 用户角色
5. 使用场景
6. 功能范围
7. 业务流程
8. 功能详细说明
9. 页面或交互说明
10. 业务规则
11. 异常场景
12. 验收标准
13. 非目标范围
14. 风险和依赖
```

不是每个项目都要写得很重。

小项目可以简化，但至少要讲清楚：

```text
为什么做
做什么
怎么用
边界是什么
验收标准是什么
```

## 4. 产品文档模板

下面是一份通用产品文档模板。

````markdown
# 产品需求文档：功能名称

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 需求名称 |  |
| 所属项目 |  |
| 产品负责人 |  |
| 技术负责人 |  |
| 创建时间 |  |
| 当前状态 | 草稿 / 评审中 / 开发中 / 已上线 |

## 2. 需求背景

说明为什么要做这个需求。

例如：

- 当前用户遇到了什么问题。
- 当前系统有什么不足。
- 业务上为什么需要这个能力。

## 3. 目标

说明这个需求希望达到什么结果。

例如：

- 用户可以更方便地管理任务。
- 后端可以支持按条件查询。
- 降低用户查找任务的成本。

## 4. 用户角色

说明谁会使用这个功能。

| 角色 | 说明 |
| --- | --- |
| 普通用户 | 创建、查看、修改自己的任务 |
| 管理员 | 管理系统中的任务数据 |

## 5. 使用场景

描述真实使用场景。

例如：

```text
用户有很多待办任务，希望按照工作、学习、生活分类管理。
```

## 6. 功能范围

本次要做：

- 功能 A。
- 功能 B。
- 功能 C。

本次不做：

- 功能 X。
- 功能 Y。

## 7. 业务流程

用文字说明用户怎么完成操作。

```text
用户进入任务列表
用户点击创建任务
用户填写任务标题和内容
用户提交
系统保存任务
系统返回创建成功
```

## 8. 功能详细说明

### 8.1 功能点一

说明：

- 输入什么。
- 输出什么。
- 有哪些限制。
- 有哪些状态。

### 8.2 功能点二

说明同上。

## 9. 页面或交互说明

如果是前端功能，说明页面布局、按钮、弹窗、提示文案等。

如果是纯后端 API 项目，可以写接口使用方式或请求示例。

## 10. 业务规则

例如：

- 任务标题不能为空。
- 任务状态只能是 `pending`、`doing`、`done`。
- 删除任务后不再出现在任务列表。

## 11. 异常场景

例如：

- 参数为空。
- 数据不存在。
- 用户没有权限。
- 系统内部错误。

## 12. 验收标准

用可验证的方式描述完成标准。

- 能创建任务。
- 能查询任务列表。
- 能按状态筛选任务。
- 参数错误时返回明确错误。
- 删除任务后再次查询返回不存在。

## 13. 风险和依赖

例如：

- 是否依赖其他系统。
- 是否需要前端配合。
- 是否需要数据库变更。
- 是否影响旧数据。
````

## 5. 技术设计文档是什么

技术设计文档通常也叫 TDD。

这里的 TDD 不是测试驱动开发，而是：

```text
Technical Design Document
技术设计文档
```

它主要回答：

```text
这个需求怎么实现？
系统要改哪些地方？
接口怎么设计？
数据怎么存？
流程怎么走？
有哪些风险和影响？
```

技术设计文档一般在需求评审之后、正式开发之前写。

它的作用是：

- 让开发前先想清楚方案。
- 让别人能评审你的技术方案。
- 降低开发中返工。
- 方便测试理解影响范围。
- 方便后续维护者理解当时为什么这样设计。

## 6. 技术设计文档应该包含什么

你提到的内容很完整，一份技术设计文档建议至少包含：

```text
1. 技术设计背景
2. 需求范围
3. 总体方案
4. 接口文档
5. 接口内部流程
6. 数据库设计文档
7. 整体数据流程
8. 数据库 ER 图
9. 预估影响范围
10. 本次新增或修改的配置文件
11. 异常处理
12. 测试建议
13. 上线和回滚方案
```

如果是复杂项目，还可以增加：

```text
权限设计
缓存设计
消息队列设计
定时任务设计
幂等设计
性能评估
安全设计
兼容性设计
灰度方案
监控告警
```

````markdown
## 5. 接口文档

### 5.1 接口名称

```text

```

#### 请求说明

| 项目 | 内容 |
| --- | --- |
| 请求方法 | POST |
| 请求路径 | `/api/todos` |
| Content-Type | `application/json` |
| 说明 | 创建任务 |

| 字段 | 类型 | 是否必填 | 说明 |

```json

  "content": "完成接口开发",

```

#### 响应参数

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | number | 任务 ID |
| title | string | 任务标题 |
| content | string | 任务内容 |
| status | string | 任务状态 |

#### 响应示例

```json
{
  "id": 1,
  "title": "学习 Gin",
  "content": "完成接口开发",
  "status": "pending"
}
```

#### 错误码

| HTTP 状态码 | 场景 | 说明 |
| --- | --- | --- |
| 400 | 参数错误 | 请求参数不合法 |

## 6. 接口内部流程

说明一个接口从接收请求到返回响应，中间经过哪些步骤。

```text
1. Gin 接收 HTTP 请求
2. Router 匹配接口路径
3. Controller 解析请求参数
4. Controller 校验参数
5. Controller 调用 Service
6. Service 执行业务逻辑
7. Service 使用 Gorm 操作数据库
8. 数据库返回执行结果
9. Service 返回业务结果
10. Controller 组装 JSON 响应
11. Gin 返回 HTTP 响应
```
````

## 8. 产品文档和技术设计文档的区别

| 对比项 | 产品文档 | 技术设计文档 |
| --- | --- | --- |
| 核心问题 | 做什么，为什么做 | 怎么做，怎么实现 |
| 主要读者 | 产品、研发、测试、业务方 | 研发、测试、架构、运维 |
| 关注重点 | 用户场景、业务规则、验收标准 | 接口、流程、数据库、影响范围 |
| 是否写代码细节 | 通常不写 | 需要写关键实现方案 |
| 是否写接口 | 可以简单写 | 必须详细写 |
| 是否写数据库 | 通常不写或简写 | 必须详细写 |

一句话区分：

```text
产品文档告诉团队：我们要建一座什么样的桥。
技术设计文档告诉团队：这座桥用什么材料、什么结构、怎么施工、会影响哪里。
```

## 9. TaskNest 示例：新增任务分类功能

下面用 TaskNest 举一个简化示例。

### 9.1 产品文档简版

````markdown
# 产品需求文档：任务分类功能

## 1. 需求背景

当前 TaskNest 只支持创建任务，但任务多了以后无法分类管理。
用户希望按照工作、学习、生活等分类查看任务。

## 2. 目标

- 用户可以创建任务分类。
- 用户可以给任务绑定分类。
- 用户可以按分类筛选任务。

## 3. 功能范围

本次包含：

- 创建分类。
- 查询分类列表。
- 创建任务时选择分类。
- 查询任务时按分类筛选。

本次不包含：

- 分类排序。
- 分类图标。
- 多级分类。

## 4. 业务规则

- 分类名称不能为空。
- 分类名称长度不能超过 30 个字符。
- 删除分类前，如果分类下还有任务，不允许删除。
- 任务可以没有分类。

## 5. 验收标准

- 能成功创建分类。
- 能查询分类列表。
- 创建任务时可以传入分类 ID。
- 查询任务列表时可以按分类 ID 筛选。
- 分类不存在时创建任务返回错误。
````

### 9.2 技术设计文档简版

````markdown
# 技术设计文档：任务分类功能

## 1. 技术设计背景

当前 TaskNest 的 Todo 只有标题、内容和状态。
为了支持任务分类，需要新增分类表，并在任务表中增加分类关联字段。

## 2. 总体方案

- 新增 `categories` 表。
- `todos` 表新增 `category_id` 字段。
- 新增分类相关接口。
- 修改任务创建接口，支持传入 `category_id`。
- 修改任务列表接口，支持按 `category_id` 查询。

## 3. 接口文档

### 3.1 创建分类

```text
POST /api/categories
```

请求：

```json
{
  "name": "学习"
}
```

响应：

```json
{
  "id": 1,
  "name": "学习"
}
```

### 3.2 查询分类列表

```text
GET /api/categories
```

### 3.3 创建任务时绑定分类

```text
POST /api/todos
```

请求：

```json
{
  "title": "学习 Gorm",
  "content": "理解表关系",
  "category_id": 1
}
```

### 3.4 按分类查询任务

```text
GET /api/todos?category_id=1
```

## 4. 接口内部流程

### 创建任务

```text
1. Controller 读取请求体
2. 校验 title 是否为空
3. 如果传入 category_id，查询分类是否存在
4. Service 创建 Todo
5. Gorm 写入 todos 表
6. 返回创建后的任务数据
```

## 5. 数据库设计

### categories 表

### todos 表修改

  |
  v

  |
  v

  |
  v

```

## 7. 数据库 ER 图

```text
+------------+        +---------+
| categories | 1    N | todos   |
+------------+--------+---------+
| id         |        | id      |
| name       |        | title   |
| created_at |        | category_id |
| updated_at |        | status  |
+------------+        +---------+
```

## 8. 预估影响范围

| 类型 | 影响范围 | 说明 |
| --- | --- | --- |
| 接口 | `/api/todos` | 创建和查询任务接口需要改 |
| 新接口 | `/api/categories` | 新增分类接口 |

本次无新增或修改配置文件。
````

## 10. 写文档时最容易漏掉什么

新手写技术设计文档时，常见遗漏是：

- 只写接口，不写内部流程。
- 只写新增内容，不写对旧接口的影响。
- 只写正常情况，不写异常情况。
- 只写数据库字段，不写字段是否允许为空。
- 只写开发方案，不写测试建议。
- 只写上线步骤，不写回滚方案。
- 忘记说明有没有配置文件变更。

写完后可以用下面的问题自查：

```text
别人只看这份文档，能不能知道要做什么？
后端能不能按这份文档开发？
测试能不能按这份文档写用例？
上线的人能不能知道影响范围？
出了问题能不能知道怎么回滚？
```
